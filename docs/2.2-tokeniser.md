# 2.2 BPE tokeniser (`src/tokenizer.py`)

This script trains a **Byte-Pair Encoding (BPE)** subword tokeniser on the **training split only**, encodes all three splits, pads sequences to a fixed length, and saves NumPy tensors plus label encodings for the neural authorship model (CNN-LSTM).

## Purpose

isiZulu is agglutinative: meaningful style and morphology often live in prefixes and suffixes (e.g. `uku-`, `ama-`, `nga-`) rather than in whole-word tokens alone. BPE learns frequent character merges from the corpus so the downstream model can represent subword patterns without a hand-built morphological analyser.

Training the tokeniser **only on train** avoids vocabulary leakage from validation and test chunks.

## Prerequisites

Run [`src/preprocessing.py`](2.1-preprocessing.md) first so these files exist:

- `data/splits/train.csv`
- `data/splits/val.csv`
- `data/splits/test.csv`

## Hyperparameters

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `VOCAB_SIZE` | 3000 | Small enough for a low-resource corpus; encourages subword rather than whole-word memorisation |
| `MIN_FREQUENCY` | 2 | Drop hapax merges from training |
| `PERCENTILE` | 95 | Set max sequence length from 95th percentile of **train** lengths (limits padding from outliers) |
| `RANDOM_STATE` | 42 | Label encoding reproducibility |

Special tokens: `[UNK]` (unknown subwords), `[PAD]` (padding). Pre-tokeniser: **Whitespace** (word boundaries preserved before BPE merges).

## Pipeline overview

```
train.csv ──► train BPE ──► tokenizer.json
                │
                ├── encode train / val / test
                ├── choose max_seq_len (95th pct on train)
                ├── pad / truncate → X_train, X_val, X_test
                ├── LabelEncoder on author → y_train, y_val, y_test
                └── evaluate + save metadata
```

## Main functions

| Function | Role |
|----------|------|
| `load_splits` | Loads the three CSV splits |
| `train_bpe_tokenizer` | Fits BPE on training texts only via Hugging Face `tokenizers` |
| `save_tokenizer` | Writes `outputs/bpe_tokenizer/tokenizer.json` |
| `encode_split` | Maps each chunk to a variable-length list of token IDs |
| `analyse_lengths` | Computes length stats; sets `max_sequence_length`; saves histogram |
| `pad_and_truncate` | Builds fixed-size `(n_samples, max_sequence_length)` arrays; pads with `[PAD]` id |
| `encode_labels` | `sklearn.preprocessing.LabelEncoder` fit on train authors |
| `inspect_bpe_segmentations` | Prints example subword splits for manual QA |
| `evaluate_tokenizer` | UNK rate, fertility, truncation rate, vocabulary coverage |
| `save_arrays` | Writes `.npy` files, `label_classes.json`, `tokenizer_metadata.json` |
| `run` | Orchestrates the full tokenisation pipeline |

### Padding detail

Arrays are pre-filled with the padding token ID (`1` for `[PAD]` in the current run). Sequences longer than `max_sequence_length` are truncated from the start (first `max_sequence_length` tokens kept). The embedding layer treats pad IDs as real inputs; the model relies on fixed-length batches rather than a separate masking layer for padding.

## Design choices

1. **Train-only BPE**: Same principle as fitting TF-IDF only on train: test vocabulary must not influence token merge statistics.
2. **Percentile-based max length**: Using the absolute maximum length would force very long padding for a few outlier chunks; the 95th percentile balances truncation vs. efficiency.
3. **Small vocabulary (3000)**: On a low-resource corpus, a large vocab encourages sparsity and overfitting to surface forms.
